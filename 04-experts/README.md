# WorkBuddy 专家与专家团实践指南

## TL;DR

专家适合把 WorkBuddy 临时切换成某个领域角色，用更稳定的方法论处理单点问题。专家团适合需要多角色拆解、并行执行和整合交付的复杂任务。它们不是系统权限开关；真正涉及文件、外部服务或脚本执行时，风险通常来自配套的 Skill、MCP 或 Connector。

## Official facts

- WorkBuddy 官方文档将专家描述为用于配置专业角色，在特定领域中以更明确的方法和视角完成任务。
- 专家板块包含“专家中心”和“我的专家”，支持浏览专家和专家团，也支持创建属于自己的专家。
- 官方说明中，专家是“角色切换”机制，以人设、方法论和工具链让 WorkBuddy 用特定领域专家身份执行任务。
- 专家团是“协作执行”机制，由多位专家分工协作，团长会自动拆解、并行执行并整合交付。
- 官方对 Skill、专家、专家团的区分是：Skill 是工具能力，专家是懂某领域的 AI 角色，专家团是多位专家加协作流程。
- 官方提示专家本身不主动获取系统权限；只有配备 Skill / MCP 时，才会在用户授权下间接访问文件或外部服务。

## Practical interpretation

专家的价值不在于“更像真人”，而在于把任务处理方式固定下来：同一类问题每次都采用相近的判断框架、输入清单和输出格式。专家团的价值是把多视角评审变成一个工作流，例如“产品经理拆需求、法务看风险、运营写落地计划、项目经理整合排期”。

不要把专家当成万能审批人。专家输出仍然是模型生成内容，适合辅助分析、草拟方案、形成检查清单；涉及医疗、法律、投资、合规、财务和人事处分等重大事项时，应由具备资质或授权的人复核。

## How to use

1. 先判断任务类型：单领域问题选专家，多领域交付选专家团。
2. 在任务说明中写清业务背景、输入材料、输出格式、验收标准和禁止事项。
3. 让专家先给出处理计划，再执行正文输出。
4. 对专家团任务，要求输出角色分工、各角色结论、整合后的最终建议。
5. 如任务涉及 Skill、Connector、文件读写或外部服务，先单独确认权限和数据范围。

## Best practices

- 用专家处理“标准化判断”，例如文案审校、报告结构优化、招聘 JD 评审、会议纪要整理。
- 用专家团处理“多角色协作”，例如活动方案、产品需求、项目复盘、企业制度草案。
- 给专家明确输入材料，不要只写“你是某某专家，帮我看看”。
- 要求输出可验收产物：修改稿、差异说明、风险清单、决策表、行动项。
- 对高风险场景增加“非专业意见，仅供内部讨论”的边界说明。
- 自定义专家时，把角色、适用场景、输入要求、输出格式、限制条件写成卡片，而不是只写人设。

## Common mistakes

- 把专家身份当成事实来源，忽略官方文档、企业制度或真实业务数据。
- 对专家团只输入一句宽泛需求，导致拆解方向漂移。
- 让多个专家重复做同一件事，而不是明确分工。
- 忽略积分消耗；专家团通常涉及多位专家和多轮交互，成本高于单专家。
- 误以为专家会自动拥有文件、系统或第三方服务权限。

## Security / permission notes

专家本身主要处理用户主动提供的对话和上传文件。风险边界在以下情况会扩大：

- 专家绑定或调用 Skill / MCP。
- 专家任务要求读取、写入、删除或整理文件。
- 专家任务调用外部服务、API、邮箱、会议、文档或知识库。
- 用户上传包含个人信息、商业秘密、合同、财务数据的材料。

建议在专家任务前加一句约束：“如需调用 Skill、Connector、MCP、脚本或访问工作区文件，请先列出用途、范围和风险，等待确认后再执行。”
