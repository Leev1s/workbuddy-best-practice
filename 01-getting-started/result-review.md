# WorkBuddy 结果验收指南

## TL;DR

WorkBuddy 的任务完成不等于业务完成。用户需要检查产物、全部文件、变更和预览，确认目标、输入、输出、路径、权限和外部调用都符合预期。结果验收是把 Agent 输出变成可交付成果的关键步骤。

## Official facts

- 官方结果查看页面说明，右侧结果区用于查看 WorkBuddy 产出的文件、代码变更和预览内容。
- 结果区主要提供四类内容：产物、全部文件、变更和预览。
- 产物适合查看任务生成的文档、表格、PPT、分析报告等交付文件。
- 全部文件用于查看工作空间中的文件内容和文件位置。
- 变更用于查看本次任务带来的文件修改。
- 预览用于查看网页、页面原型或可直接打开的结果页面。

## Practical interpretation

验收不是“看起来还行”，而是回答七个问题：

1. 是否满足任务目标？
2. 是否使用了正确输入？
3. 是否输出到正确位置？
4. 是否有误删、覆盖或越界写入？
5. 是否存在未说明假设？
6. 是否有外部调用或数据外发？
7. 是否可以交付给他人？

## How to review

### 1. 产物：看生成了什么

检查交付物是否存在、格式是否正确、命名是否清晰、内容是否完整。对报告、PPT、表格、Markdown、PDF 等文件，确认它们能被目标读者直接使用。

### 2. 全部文件：看文件位置和内容

检查文件是否写入约定目录，是否产生了意外文件，原始输入是否被改动。对批量任务，随机抽查样本并确认清单覆盖所有文件。

### 3. 变更：看修改了哪些文件

检查每个改动是否必要、路径是否正确、是否覆盖了重要文件。涉及脚本、配置、批量重命名或删除计划时，应优先看变更。

### 4. 预览：看最终效果

对网页、文档、表格、PPT 或可视化结果，使用预览检查排版、图表、链接、交互和可读性。不要只相信文字总结。

## Review checklist

- [ ] 是否满足原始目标？
- [ ] 是否覆盖所有指定输入？
- [ ] 是否正确读取 prompt 中指定的 `.workbuddy/` 项目级上下文文件？
- [ ] 是否使用了正确的数据口径和时间范围？
- [ ] 是否写入正确目录？
- [ ] 是否误把临时产物、日志、缓存或私人上下文写入 `.workbuddy/`？
- [ ] 是否保留原始文件或使用副本？
- [ ] 是否存在误删、覆盖、移动或越界写入风险？
- [ ] 是否误改 `.workbuddy/` 中的项目规则、术语表、review checklist 或安全说明？
- [ ] 是否有未说明假设、缺失字段或待确认事项？
- [ ] 是否有外部调用、联网、API 请求或数据外发？
- [ ] 是否满足格式、命名、页数、语言和受众要求？
- [ ] 是否可以直接交付给同事、客户或管理层？
- [ ] 是否需要人工二次审批？
- [ ] 是否值得沉淀为 workflow template？

## Best practices

- 对高风险任务先看变更，再看产物。
- 对数据分析先核对输入覆盖率，再看结论。
- 对文档生成先看结构和事实，再看文风。
- 对批量文件处理先验收清单，再允许执行。
- 对自动化候选任务记录失败条件和回滚方式。

## Common mistakes

- 只看最后回答，不打开产物。
- 只看产物，不看全部文件和变更。
- 忽略未确认假设，把推测当事实。
- 未检查敏感数据是否进入第三方服务。
- 自动化前没有做多轮人工验收。

## Security / permission notes

如果发现任务涉及删除文件、覆盖文件、执行脚本、调用外部程序、访问网络、写入工作空间外目录或处理敏感数据，应暂停交付，要求 WorkBuddy 解释操作理由、影响范围和替代方案。
